# 训练服务器操作手册（租用 3090 · 从零到出货）

面向：一台新租的带 3090（或同级）的 Linux GPU 服务器。
目标：**训练三个 LoRA 专家 → 转成 GGUF → 拿到可在用户端侧运行的权重**。

全程约 1~2 小时（大头是环境安装和下载）。每一步都可直接复制执行，`<...>` 为需替换的占位符。

> 术语：**本地** = 你自己的电脑；**服务器** = 租的 3090 机器。
> base 默认 `Qwen2.5-Coder-1.5B-Instruct`（端侧友好）；要 3B 就把命令里的 `1.5B` 换成 `3B`。

---

## 阶段总览

| 步骤 | 在哪做 | 产物 |
|---|---|---|
| 0 准备数据 | 本地或服务器 | `data/*.jsonl` |
| 1 连服务器 + 装环境 | 服务器 | LLaMA-Factory 可用 |
| 2 上传代码与数据 | 本地→服务器 | 仓库 + 数据在服务器 |
| 3 训练三个专家 | 服务器 | `adapters/{planner,executor,debugger}` |
| 4 转 GGUF | 服务器 | `adapters/*-f16.gguf` + `base.gguf` |
| 5 服务器上验证热切换 | 服务器 | 确认三专家行为不同 |
| 6 打包回本地 | 服务器→本地 | 端侧出货权重（~1.5GB） |

---

## 步骤 0 · 准备训练数据

数据生成**不需要 GPU**，两种做法二选一：

**A. 本地先生成好再上传**（推荐，省服务器时长费用）
```bash
# 本地，slmagent 环境
cd data_gen
python run_all.py --per-category 40 --workers 8
# 产出 data/planner.jsonl / executor.jsonl / debugger.jsonl
```

**B. 到服务器上再生成**（服务器能连 DeepSeek 时）——见步骤 2 之后执行同样命令。

确认三份 jsonl 都非空、格式正确：
```bash
wc -l data/*.jsonl
head -n1 data/planner.jsonl        # 应是含 instruction/input/output 的 JSON
```

---

## 步骤 1 · 连上服务器并准备环境

### 1.1 连接 + 开 tmux（防断线丢训练）
```bash
ssh <user>@<server-ip>
tmux new -s train          # 断线后用 tmux attach -t train 回到现场
nvidia-smi                 # 确认能看到 3090 及显存(应 ~24GB)
```

### 1.2 建 Python 环境
```bash
# 若有 conda：
conda create -n leap python=3.10 -y && conda activate leap
# 或用 venv：
# python3 -m venv ~/leap && source ~/leap/bin/activate
```

### 1.3 安装 LLaMA-Factory（含 QLoRA 依赖）
```bash
pip install --upgrade pip
pip install "llamafactory[torch,metrics,bitsandbytes]"
# bitsandbytes 是 4-bit QLoRA 必需；装不上见步骤 7
llamafactory-cli version   # 能打印版本即安装成功
```

### 1.4 配国内 HF 镜像（下载模型快很多）
```bash
pip install -U huggingface_hub
export HF_ENDPOINT=https://hf-mirror.com     # 建议加进 ~/.bashrc
```

## 步骤 2 · 把代码和数据传上服务器

在**本地**仓库根目录执行（用 rsync，`data/` 里的 jsonl 一并带上）：
```bash
rsync -avz --exclude '.git' --exclude 'node_modules' --exclude '__pycache__' \
  ./ <user>@<server-ip>:~/SmallCLI/
```
> 若走 git：服务器 `git clone` 你的仓库，再单独把 `data/*.jsonl`（.gitignore 忽略了）rsync 上去。

服务器上确认结构：
```bash
cd ~/SmallCLI
ls train/                       # planner.yaml executor.yaml debugger.yaml
ls data/                        # dataset_info.json + 三份 jsonl
cat data/dataset_info.json      # 确认注册了 leap_planner/executor/debugger
```
（若选步骤 0-B，在此 `pip install openai python-dotenv`，配好 `.env`，`cd data_gen && python run_all.py ...`）

---

## 步骤 3 · 训练三个专家

**在仓库根目录**执行（配置里 `dataset_dir: data` 是相对根目录的）：
```bash
cd ~/SmallCLI
llamafactory-cli train train/planner.yaml
llamafactory-cli train train/executor.yaml
llamafactory-cli train train/debugger.yaml
```
- 1.5B + QLoRA，单张 3090，每个专家约 **5~20 分钟**（取决于数据量）。
- 三条串行跑即可（低并发，不必并行抢显存）。
- 训练时关注日志里的 `loss` 应稳定下降；结束后各目录有 `adapter_model.safetensors` 和 `training_loss.png`。

产物：
```
adapters/planner/      adapters/executor/      adapters/debugger/
```

### 3.1 快速抽查单个专家是否学到东西
```bash
llamafactory-cli chat \
  --model_name_or_path Qwen/Qwen2.5-Coder-1.5B-Instruct \
  --adapter_name_or_path adapters/executor \
  --template qwen
# 输入一句“查看当前目录文件”，应回一个命令 list，如 ["ls"]
```
三个专家分别抽查一句，行为明显不同即成功（planner 出步骤 list、executor 出命令、debugger 出修正命令）。

> **显存不够（OOM）**：编辑对应 yaml，把 `per_device_train_batch_size: 1`、`cutoff_len: 1536`，重跑。

## 步骤 4 · 转成 GGUF（端侧用）

### 4.1 编译 llama.cpp + 拿转换脚本
```bash
cd ~
git clone https://github.com/ggml-org/llama.cpp
cd llama.cpp
cmake -B build -DGGML_CUDA=ON && cmake --build build -j    # 服务器有卡，开 CUDA 顺便验证
pip install -r requirements.txt                            # convert_lora_to_gguf.py 依赖
```

### 4.2 下载量化好的 base GGUF
```bash
cd ~/SmallCLI
huggingface-cli download Qwen/Qwen2.5-Coder-1.5B-Instruct-GGUF \
  qwen2.5-coder-1.5b-instruct-q4_k_m.gguf --local-dir models/gguf
```

### 4.3 三个 adapter：HF → GGUF
```bash
cd ~/SmallCLI
for role in planner executor debugger; do
  python ~/llama.cpp/convert_lora_to_gguf.py adapters/$role \
    --base Qwen/Qwen2.5-Coder-1.5B-Instruct \
    --outtype f16 --outfile adapters/$role-f16.gguf
done
ls -lh adapters/*-f16.gguf      # 每个应只有几~几十 MB
```
> `--base` 指向训练用的同一个 HF base（会自动下载其 config/权重以对齐张量）。

## 步骤 5 · 服务器上先验证「热切换」（可选但强烈建议）

先在服务器把整条链路验通，再拷回本地，避免端侧才发现问题。
```bash
~/llama.cpp/build/bin/llama-server \
  -m models/gguf/qwen2.5-coder-1.5b-instruct-q4_k_m.gguf \
  --lora adapters/planner-f16.gguf \      # id 0
  --lora adapters/executor-f16.gguf \     # id 1
  --lora adapters/debugger-f16.gguf \     # id 2
  -c 4096 --port 8001
```
另开一个终端，用 `lora` scale 分别激活三个专家，确认输出风格不同：
```bash
# 激活 executor(id=1)，其余 scale=0
curl -s http://localhost:8001/v1/chat/completions -H 'Content-Type: application/json' -d '{
  "messages":[{"role":"system","content":"指令：将描述转换为Linux命令，用python list形式输出"},
              {"role":"user","content":"查看当前目录文件"}],
  "lora":[{"id":0,"scale":0},{"id":1,"scale":1},{"id":2,"scale":0}]
}'
```
把 `id=0` 设 1、其余 0，换 planner 的 system/prompt，应改出「步骤 list」——证明热切换生效。

---

## 步骤 6 · 打包回本地 / 出货

端侧只需要 **1 个 base GGUF + 3 个 adapter GGUF**：
```bash
# 在本地执行，把出货权重拉回来
rsync -avz <user>@<server-ip>:~/SmallCLI/models/gguf/qwen2.5-coder-1.5b-instruct-q4_k_m.gguf ./models/gguf/
rsync -avz <user>@<server-ip>:~/SmallCLI/adapters/'*-f16.gguf' ./adapters/
```
总体积约 **1~1.5GB**（base ~1GB + 三个 adapter 各几十 MB）。
之后端侧启动方式见 `train/README.md` Stage 3 / `docs/技术路线与实施计划.md` 阶段 3。

**关掉计费**：确认权重都拷回本地、能正常加载后，再释放/关停服务器。

## 步骤 7 · 常见问题

| 现象 | 处理 |
|---|---|
| `bitsandbytes` 装不上/报 CUDA | 确认 `nvidia-smi` 驱动 OK；`pip install bitsandbytes>=0.43`；仍不行可临时去掉 yaml 里 `quantization_bit: 4` 改用普通 LoRA（1.5B 显存够） |
| HF 下载极慢/失败 | 已 `export HF_ENDPOINT=https://hf-mirror.com`；大文件用 `huggingface-cli download` 而非 git |
| 训练 loss 不降/乱码 | 检查 `template: qwen` 没写错；抽查 `data/*.jsonl` 格式；学习率过大可降到 `5e-5` |
| OOM | `per_device_train_batch_size: 1`、`cutoff_len: 1536`、`gradient_accumulation_steps` 调大补 batch |
| `convert_lora_to_gguf.py` 张量不匹配 | `--base` 必须与训练时 `model_name_or_path` 完全一致（同为 1.5B-Instruct） |
| 断线训练中断 | 全程在 `tmux` 里跑；重连 `tmux attach -t train` |
| 想换 3B | 三个 yaml 的 `model_name_or_path` 改 `Qwen2.5-Coder-3B-Instruct`，步骤 4 的 GGUF base 与 `--base` 同步换 3B |

## 一页速查（全部命令）
```bash
# 1. 环境
ssh <user>@<server-ip>; tmux new -s train
conda create -n leap python=3.10 -y && conda activate leap
pip install "llamafactory[torch,metrics,bitsandbytes]"
export HF_ENDPOINT=https://hf-mirror.com
# 2. 传数据(本地执行)
rsync -avz --exclude '.git' ./ <user>@<server-ip>:~/SmallCLI/
# 3. 训练(服务器, 仓库根目录)
cd ~/SmallCLI
llamafactory-cli train train/planner.yaml
llamafactory-cli train train/executor.yaml
llamafactory-cli train train/debugger.yaml
# 4. 转 GGUF
git clone https://github.com/ggml-org/llama.cpp ~/llama.cpp
cmake -S ~/llama.cpp -B ~/llama.cpp/build -DGGML_CUDA=ON && cmake --build ~/llama.cpp/build -j
pip install -r ~/llama.cpp/requirements.txt
huggingface-cli download Qwen/Qwen2.5-Coder-1.5B-Instruct-GGUF qwen2.5-coder-1.5b-instruct-q4_k_m.gguf --local-dir models/gguf
for role in planner executor debugger; do
  python ~/llama.cpp/convert_lora_to_gguf.py adapters/$role --base Qwen/Qwen2.5-Coder-1.5B-Instruct --outtype f16 --outfile adapters/$role-f16.gguf
done
# 5. 拷回本地(本地执行) → 关服务器
```
