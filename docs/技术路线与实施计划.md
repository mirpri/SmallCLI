# LEAP 技术路线与实施计划

> 大创项目 · 目标：快速落地一个「单冻结底座 + 三 LoRA 专家热切换」的轻量化 CLI Agent，
> 并给出相对 Baseline 的可量化优势。对应 `LEAP.md` 架构与 `应用场景及定位分析.md` 交付物。

---

## 1. 锁定的技术路线

| 环节 | 选定方案 | 理由 |
|---|---|---|
| Base 模型 | **Qwen2.5-Coder-1.5B-Instruct**（端侧默认，单一、冻结，三专家共用）；3B 作高配档 | 端侧真正约束是用户 CPU 推理速度，1.5B Q4≈1GB 在无显卡笔记本可跑；共用同一 base 是热切换前提。base 尺寸按端侧目标机定，不按训练算力定 |
| 微调方法 | **QLoRA**（4-bit + LoRA, r=16 / α=32） | 单张 3090（24GB）可训，预算友好，符合 PEFT 前提 |
| 训练框架 | **LLaMA-Factory** | Qwen2.5 开箱即用，三专家改 config 即可 |
| 训练数据 | **用 DeepSeek-V3.2 蒸馏**（复用现有接口） | 免人工标注，是能否快速落地的命门 |
| 推理 / 热切换 | **llama.cpp（GGUF）一进程挂 3 LoRA** | 每请求 `lora` scale 切专家 = LEAP 热切换；轻量、CPU 可跑，契合端侧低并发定位（vLLM 太重、Ollama 做不到多 LoRA 热切换） |
| 编排 | **复用现有 LangGraph**（`orchestrator_langgraph.py`） | `router_node` 已按 `综合推理/执行命令/文件编辑` 分流，改挂不同 LoRA 即可 |

### 三个 LoRA 专家（对应 `LEAP.md` 专家分类）
1. **Planner** — 用户任务 → 带前缀步骤序列（`综合推理: / 执行命令: / 文件编辑:`）
2. **Executor** — 自然语言描述 → shell 命令 list
3. **Debugger** — 命令 + 报错 → 修正后命令

## 2. 明确砍掉 / 降级的部分（风险控制）

`LEAP.md` 3.2 的 **「竞争性并行 / 权重叠加」** $W=\alpha W_{exec}+\beta W_{debug}$：

- 两个独立训练的 LoRA 直接加权求和，**数学上不保证得到有意义行为**（不同于联合训练的 MoE）；
- llama.cpp 的 `lora` scale 虽可对多个 adapter 同时给非零权重（形式上做到叠加），但同上，**行为无保证**，不作为主线。

**处理**：从主线移除，降级为**路由层集成**（router 同时问 Executor 与 Debugger，再择一），
作为论文的 **future work / 消融实验**。主线「单 base + 3 LoRA 热切换」已足够新颖、可落地。

## 3. 分阶段实施计划（约 7–10 天）

### 阶段 0 · 统一底座（0.5 天）
- 把 `slm1/2/3` 中混用的 `qwen2.5:3b` 与 `qwen2.5-coder:1.5b` 统一为 `Qwen2.5-Coder-1.5B-Instruct`（端侧默认）。
- 纯 prompt 跑通 = **Baseline A**，同时作为回归基准。

### 阶段 1 · DeepSeek 蒸馏数据管线（2–3 天）✅ 代码已完成
- 产物：`data/{planner,executor,debugger}.jsonl`（LLaMA-Factory `alpaca` 格式）。
- 关键设计：训练/推理共用 prompt 模板；管线自洽（Planner 子步 → Executor → Debugger）；逐条输出校验。
- **不依赖 GPU**，在本地 conda 环境即可运行。

### 阶段 2 · QLoRA 训练三专家（1 天）✅ 配置已就绪
- 在 3090 上：`llamafactory-cli train train/{planner,executor,debugger}.yaml`（详见 `docs/训练服务器操作手册.md`）。
- 产物：`adapters/{planner,executor,debugger}`。单个 adapter 约 10–30 分钟。

### 阶段 3 · llama.cpp 热切换 + 接回 LangGraph（1–2 天）
- adapter `convert_lora_to_gguf.py` 转 GGUF；`llama-server -m <base.gguf> --lora planner --lora executor --lora debugger`（一进程挂 3 LoRA，id 0/1/2）。
- 把 `orchestrator_langgraph.py` 各节点的 `ollama.generate(...)` 换成 OpenAI 兼容调用，用 `extra_body={"lora":[...]}` 把目标专家 scale 设 1.0、其余 0.0 → **LEAP 系统跑通**。
- 详见 `train/README.md` Stage 3。

### 阶段 4 · 评测（1–2 天）
- 攒 50–100 条 CLI 任务基准。
- 对比 **Baseline A（prompt SLM）/ Baseline C（DeepSeek prompt）/ LEAP**。
- 指标：任务成功率、平均延迟、Token 消耗。Baseline B（全量微调）时间不够可跳过。

## 7. 关键风险与对策

| 风险 | 对策 |
|---|---|
| 蒸馏数据质量参差 | 逐条 `ast.literal_eval` + 前缀/格式校验，脏样本丢弃；抽样人工核查 |
| 1.5B 精度不足 | 升 3B（端侧高配档）或 7B（仅云端）：配置改 `model_name_or_path` 即可，其余不变 |
| Demo 部署 | 已改用 llama.cpp（Q4 量化 3B），本地 CPU/笔记本即可跑，不再强依赖 GPU；需远程时 `server.py`/前端仍可远程调用 |
| 「权重叠加」调不出 | 已降级为 future work，不阻塞主线 |
| 评测任务偏简单不可信 | 基准覆盖简单/多步/需推理任务，公开任务清单保证可复现 |
